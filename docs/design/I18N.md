# Internationalization (i18n)

## One page, many languages

Marketing and auth routes are **not duplicated per language**. There is a single page file per route, for example:

- `app/[locale]/(marketing)/page.tsx` → homepage (`CrmLanding` or `OfmMarketing` namespace depending on page component)
- `app/[locale]/(marketing)/login/page.tsx` → app login shell (`HomeBranding`, `Auth`, etc.)
- `app/[locale]/(auth)/sign-in/[[...sign-in]]/page.tsx` → sign-in

The `[locale]` segment is how **next-intl** passes the active language into that one page. It is not a separate English page vs Spanish page.

**Copy** lives in JSON message files:

- `messages/en.json`
- `messages/es.json`
- `messages/it.json`

Components call `useTranslations("HomeBranding")` or `getTranslations("Footer")` and render the right strings for the current locale.

## URLs (`localePrefix: 'as-needed'`)

| Locale | Homepage | Sign-in | App login |
|--------|----------|---------|-----------|
| English (default) | `/` | `/sign-in` | `/login` |
| Spanish | `/es` | `/es/sign-in` | `/es/login` |
| Italian | `/it` | `/it/sign-in` | `/it/login` |

English does **not** use `/en`. Legacy `/en/...` URLs are redirected to unprefixed paths by middleware.

Onboarding, dashboard, and API routes stay **without** a locale prefix (`/onboarding/consent`, etc.).

## Changing language

The footer language selector sets the **locale cookie** and navigates to the same path in the chosen language (e.g. `/` ↔ `/es` ↔ `/it`, `/login` ↔ `/it/login`). Labels: English, Español, Italiano (`lib/i18n/locales.ts`).

## Adding a language

1. Add the locale code to `locales` in `i18n/routing.ts`
2. Add `messages/<locale>.json`
3. Add a label in `lib/i18n/locales.ts`
